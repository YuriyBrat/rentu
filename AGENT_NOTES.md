# AGENT_NOTES

## Поточна архітектура парсингу

Парсинг у CRM винесений в окремий потік `LeadProperty`. Це проміжна база сирих або напівперевірених оголошень з OLX, Telegram, DIM.RIA, Facebook, ручного вводу або імпорту. Вона не є основною базою об'єктів, а працює як буфер для перевірки, дзвінків, дедуплікації та відбору.

Основний життєвий цикл запису:

1. Оголошення потрапляє в `LeadProperty` зі статусом `raw`.
2. Менеджер переглядає його на сторінці `/crm/parsing`.
3. Запис переводиться в `processing`, `called`, `qualified`, `duplicate`, `fake`, `rejected` або `moved`.
4. Після перевірки якісний запис можна перенести в основну базу `Property`.
5. При переносі створюється `Property` зі стадією `crmStage = "base"`, а у `LeadProperty` зберігається `propertyId` і статус `moved`.
6. Далі `Property` рухається через `base -> inspection -> rs -> ds -> zs -> archived`.

API парсингу працює через Next.js route handlers:

- `GET /api/crm/parsing` повертає список записів, фільтри, джерела та телефонну аналітику.
- `POST /api/crm/parsing` створює один або багато записів.
- `GET /api/crm/parsing/[id]` повертає один запис.
- `PATCH /api/crm/parsing/[id]` оновлює статус, результат дзвінка, нотатку або переносить запис в `Property`.

Телефонна аналітика рахується окремою утилітою. Вона нормалізує номер, шукає його в базі парсингу та основних об'єктах, після чого підказує тип контакту: власник, маклер, ймовірний маклер або невідомо.

## Виконані задачі

- Створена окрема модель `LeadProperty` для сирих оголошень.
- Додані поля джерела: `source`, `sourceId`, `sourceUrl`, `importedAt`, `raw`, `attrs`.
- Додані статуси обробки: `raw`, `processing`, `called`, `qualified`, `duplicate`, `fake`, `rejected`, `moved`.
- Додані поля перевірки: `reviewStatus`, `callResult`, `reviewNote`, `lastCallAt`, `assignedToEmployee`, `duplicateOf`.
- Додана прив'язка до основного об'єкта через `propertyId`.
- Доданий унікальний індекс `{ source, sourceId }` для точного дедупу по джерелу.
- Реалізований список парсингу з пошуком, фільтром по статусу та фільтром по джерелу.
- Реалізований ручний ввід нового оголошення.
- Реалізований bulk-імпорт через JSON або CSV.
- Реалізоване оновлення статусів з UI.
- Реалізована панель деталей оголошення.
- Реалізоване збереження результату дзвінка і нотатки перевірки.
- Реалізований перенос запису з парсингу в основну базу `Property`.
- При переносі створюються базові поля об'єкта, власник, рекламне посилання, фото та запис в історії роботи.
- Додана телефонна аналітика: кількість входжень номера, зв'язки з парсингом і об'єктами, підказка власник/маклер.
- Додана сторінка парсингу в CRM-навігацію.
- Додані стадії `Property.crmStage`: `base`, `inspection`, `rs`, `ds`, `zs`, `archived`.
- Додана сторінка `База`.
- Огляд визначено як внутрішню стадію/операційну подію, а не окрему базу.
- Джерело в парсингу показується як кнопка з переходом і кольором статусу.
- Для дублів додана прив'язка до нашого об'єкта через `duplicatePropertyId`.
- Виправлено ручне створення парсингу: записи без зовнішнього `sourceId` отримують локальний ID, тому `manual + null` більше не ламає унікальний індекс.
- `POST /api/crm/parsing` більше не використовує простий `insertMany`, а повертає звіт по створених, дублях і помилках.
- Підключена DIM.RIA інтеграція: пошук оголошень, dry-run, імпорт, оновлення існуючих, price history, raw payload.
- Додана форма імпорту DIM.RIA на сторінці `/crm/parsing`.
- Підключений Hunter/Reamak endpoint для догрузок з розширеною дедуплікацією по `reamakId`, siteId/siteName і sourceId.
- У `LeadProperty` додані `reamakId`, `priceHistory`, структуровані `images` з metadata.
- Reamak імпорт підтримує оновлення існуючих записів, price history і token auth через `REAMAK_IMPORT_TOKEN`.
- Фото парсингу зберігаються окремо від `Property`, а при переносі в `Property` автоматично не копіюються.

## Структурна домовленість: OperationEvent І Communication

У проєкті вже є `models/OperationEvent.js`. Його залишаємо для операційних подій, які мають значення для звітів і стадій:

- показ;
- огляд;
- зустріч;
- завдаток;
- подія зі зміною стадії;
- подія з результатом по об'єкту або клієнту.

Огляди краще довносити саме в `OperationEvent`, бо огляд - це операційний факт, а не просто нотатка.

Окремо треба створити модель `Communication` для масових контактів:

- дзвінки;
- SMS;
- месенджери;
- нотатки;
- follow-up;
- "не взяв трубку";
- "домовились передзвонити".

Комунікація має мати `type` і `tone`.

`type` - спосіб зв'язку:

- `call`;
- `sms`;
- `messenger`;
- `note`.

`tone` - характер нотатки:

- `positive` - позитивна;
- `negative` - негативна;
- `important` - важлива, синя;
- `info` - інформаційна, жовта.

Правило: дрібний контакт - `Communication`, операційний факт - `OperationEvent`.

Також уточнено, що `Огляд` не має бути окремою базою або окремим пунктом меню. Основні розділи: `Парсинг`, `База`, `Об'єкти`. Огляд - внутрішня стадія/подія для операційки та звітів.

## Зроблено Після Домовленості Про Комунікації

- Створено `models/Communication.js`.
- Додано `GET/POST /api/crm/communications`.
- Додано reusable `CommunicationDialog`.
- Додано reusable `CommunicationTimeline`.
- У парсингу праве "око" відкриває історію комунікацій.
- У рядку парсингу додано стрілку для розгортання додаткової інформації.
- Зміна статусу парсингу йде через `ParsingStatusDialog`.
- Для переводу в базу, неактуальний, зупинений, дубль або фейк вимагається комунікація.
- `OperationEvent` розширено типом `inspection`.
- Окрему сторінку `/crm/inspection` прибрано.
- Модалка рішення по парсингу має тільки фінальні рішення: дубль, фейк, база, неактуальний, зупинений. `Новий`, `В роботі`, `Продзвонено`, `Відхилено` прибрані.
- Для неактуального додано причину `Проданий нами`.
- Рядок парсингу показує дату внесення як `дд.мм.рр`, час дрібніше, а останню комунікацію як індикатор днів з годинником і tooltip точного часу.

  - `updated`
  - `duplicates`
  - `failed`
  - `items`
  - `errors`
- Додати `importBatchId` для партій.

### Етап 2. Дедуплікація і якість бази

- Ввести м'який дедуп по normalized phone + normalized address + price + square.
- Додати статус `duplicate` з прив'язкою `duplicateOf`.
- Показувати в UI, чому система підозрює дубль.
- Додати швидку дію "позначити дублем".
- Додати діалог порівняння двох схожих оголошень.

### Етап 3. Робочий процес менеджера

- Додати чергу "нові сьогодні".

### Етап 4. Інтеграції з реальними джерелами

### Етап 5. Перенос в основну базу об'єктів

- Вирівняти типи нерухомості, угоди, валюти, адреси та фото.
